Gemini Multi-Prompt TTS Player Deployment Guide

This document matches the current production architecture:
- Frontend and backend run in one Cloud Run service
- Firestore stores schedules, runs, and global rate-limit counters
- Firestore also stores manual async runs for refresh-safe manual generation
- Cloud Storage stores scheduled WAV and JSON artifacts
- Cloud Scheduler triggers due-schedule polling

========================================================================
1. Pre-Deploy Inspection
========================================================================

Every production deploy must start with a read-only inspection pass.
Do not deploy first and inspect later.

Required inspection commands, in this exact order:

1. Confirm active project and account
   - `gcloud config list --format=json`
   - `gcloud auth list --format=json`
2. Inspect existing Cloud Run services
   - `gcloud run services list --project western-creek-489616-m3 --format=json`
3. Inspect existing Cloud Scheduler jobs
   - `gcloud scheduler jobs list --location us-central1 --project western-creek-489616-m3 --format=json`
4. Inspect existing secrets
   - `gcloud secrets list --project western-creek-489616-m3 --format=json`
5. Inspect existing service accounts
   - `gcloud iam service-accounts list --project western-creek-489616-m3 --format=json`
6. Inspect existing buckets
   - `gcloud storage buckets list --project western-creek-489616-m3 --format=json`
7. Inspect Firestore database
   - `gcloud firestore databases describe --database='(default)' --project western-creek-489616-m3 --format=json`
8. Inspect Firestore composite indexes
   - `gcloud firestore indexes composite list --project western-creek-489616-m3 --format=json`

Deployment must not continue until this inspection pass is complete.

========================================================================
2. Minimal-Change Deployment Rules
========================================================================

Default rule: reuse existing GCP resources whenever they already satisfy the
app's needs.

Resource reuse policy:

- If the target Cloud Run service already exists, deploy to that service name.
- If a suitable service account already exists and already has the required
  permissions, reuse it.
- If a secret already exists, reuse the same secret name or existing reference
  rather than inventing a new name.
- If a bucket already exists and is already intended for this app, reuse it.
- If the scheduler job already exists, update it instead of creating a second
  job.
- If Firestore is present but a query fails because of missing indexes, create
  only the required index. Do not rename collections or redesign storage.

Avoid these unless truly necessary:

- Renaming Cloud Run services
- Rotating secret names only for cosmetic consistency
- Replacing a working service account with a different one
- Creating a second scheduler job for the same polling endpoint
- Creating duplicate artifact buckets

========================================================================
3. Deployment Decision Table
========================================================================

Use these rules after the inspection pass:

- Existing Cloud Run service found:
  - Deploy to the existing service.
- No Cloud Run service found:
  - Create the service with the documented service name.
- Existing scheduler job found:
  - Update the existing job in place.
- No scheduler job found:
  - Create the job once.
- Existing secret found:
  - Keep using the same secret name or same mounted env var source.
- No secret found:
  - Create it only if the app cannot run without it.
- Existing artifact bucket found:
  - Reuse it.
- No artifact bucket found:
  - Create a dedicated bucket for this app.
- Firestore query fails due to missing index:
  - Create the index only.

7. Firestore Notes
========================================================================

Collections used by the app:

- `schedules`
- `schedule_runs`
- `manual_runs`
- `rate_limits`

You also need the checked-in composite index definition:

- `firestore.indexes.json`

Known failure mode:

- If `POST /api/internal/run-due-schedules` returns `500` and Cloud Run logs show
  `FAILED_PRECONDITION: The query requires an index`, create the missing
  composite index before debugging anything else.

Current required command:

`gcloud firestore indexes composite create --project western-creek-489616-m3 --collection-group=schedules --field-config="field-path=enabled,order=ascending" --field-config="field-path=nextRunAt,order=ascending"`

8. Cloud Run Deploy Command
========================================================================

Deploy only after the inspection pass above.

If the service already exists, this command updates it in place.

gcloud run deploy gemini-multi-prompt-tts-player ^
  --source . ^
  --project western-creek-489616-m3 ^
  --region us-central1 ^
  --allow-unauthenticated ^
  --service-account 323507996614-compute@developer.gserviceaccount.com ^
  --set-secrets GEMINI_API_KEY=gemini-key:latest ^
  --env-vars-file C:\path\to\cloudrun-env.yaml ^
  --min-instances 1 ^
  --max-instances 10 ^
  --concurrency 80 ^
  --timeout 300 ^
  --port 8080

Notes:

- Do not set `PORT` in `--set-env-vars`
- `min-instances=1` reduces cold starts
- `max-instances=10` is the current scale setting used to handle concurrent scheduled work
- Do not include `PORT` in `--set-env-vars`; Cloud Run reserves it

9. Scheduler Job
========================================================================

Current internal polling endpoint:

- `POST /api/internal/run-due-schedules`

Required header:

- `X-Internal-Key: YOUR_SCHEDULER_SHARED_SECRET`

If the job already exists, update it instead of creating a duplicate.

Example create command:

gcloud scheduler jobs create http schedule-runner ^
  --project=western-creek-489616-m3 ^
  --location=us-central1 ^
  --schedule="*/5 * * * *" ^
  --time-zone="Europe/Paris" ^
  --uri="https://gemini-multi-prompt-tts-player-323507996614.us-central1.run.app/api/internal/run-due-schedules" ^
  --http-method=POST ^
  --headers="X-Internal-Key=YOUR_SCHEDULER_SHARED_SECRET,Content-Type=application/json" ^
  --message-body="{}"

Example update command:

gcloud scheduler jobs update http schedule-runner ^
  --project=western-creek-489616-m3 ^
  --location=us-central1 ^
  --schedule="*/5 * * * *" ^
  --time-zone="Europe/Paris" ^
  --uri="https://gemini-multi-prompt-tts-player-323507996614.us-central1.run.app/api/internal/run-due-schedules" ^
  --update-headers="X-Internal-Key=YOUR_SCHEDULER_SHARED_SECRET,Content-Type=application/json" ^
  --message-body="{}"

10. Security Notes
========================================================================

Current production hardening in the app:

- Admin auth uses signed `HttpOnly` cookies
- Browser write endpoints enforce same-origin CSRF checks via `Origin`/`Referer`
- Manual generation now runs server-side, so browser refresh does not interrupt in-progress manual runs
- `/api/artifacts/...` requires admin auth
- Artifact path traversal is blocked in the storage layer
- Daily global rate limits are stored in Firestore:
  - login: 100/day
  - text: 200/day
  - tts: 200/day

Limitations:

- If Firestore is unavailable, the rate limiter falls back to local memory
- That fallback is only for degraded operation, not the normal production path

========================================================================
11. Player Behavior
========================================================================

Current player behavior for scheduled runs:

- Scheduled audio is prefetched from top to bottom
- Prefetch is serial: one item at a time
- Each item shows a clear state:
  - `Queued`
  - `Downloading audio...`
  - `Decoding audio...`
  - `Cached locally`
  - `Failed`
- Cached scheduled audio and text are persisted locally in IndexedDB until manually cleared
